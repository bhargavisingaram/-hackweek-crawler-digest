# Crawler Digest

## The pitch

AI crawlers are scraping Toronto Life articles every day. Cloudflare sees the traffic; nobody else does. This is a prototype **weekly digest** — top scrapers, top sections, top articles, top authors — designed to land in Slack so editors and leadership can finally see how their work is being consumed by AI. Foundation for a real content-protection strategy: pay-per-crawl, robots.txt enforcement, legal evidence.

Scoped to torontolife.com for the demo. 

## Status

Runs against **real Cloudflare Analytics data** via a read-only API token scoped to one zone. Article hit counts, bot identifications, and WoW deltas are all real, from your production CF dashboard. Article titles + bylines were bootstrapped once into `data/articles.json` — see [Article metadata](#article-metadata) below.

Author names are anonymized in the digest output by default (`Writer A`, `Writer B`, …) — see `ANONYMIZE_AUTHORS` in `src/aggregate.py`. Flip to `False` once the affected writers have explicitly opted in to being named on a slide.

## Run

Set up the environment once:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Create `.env` (gitignored) with your Cloudflare credentials:

```
CLOUDFLARE_API_TOKEN=<read-only Analytics token, scoped to torontolife.com>
CLOUDFLARE_ZONE_ID=<32-char hex zone id>
```

Run the pipeline:

```bash
python src/fetch_cf_data.py     # pulls real CF analytics → data/cf_events.json
python src/aggregate.py          # prints the weekly digest
```

For an offline demo against synthetic data (no CF token required):

```bash
python src/generate_synthetic_data.py
python src/aggregate.py
```

## Layout

- `src/fetch_cf_data.py` — pulls real AI-bot traffic from Cloudflare GraphQL Analytics
- `src/generate_synthetic_data.py` — produces synthetic CF data (offline / fallback mode)
- `src/aggregate.py` — turns CF events + article metadata into the weekly digest
- `src/fetch_titles.py` — **bootstrap-only.** Pulled article titles + bylines from torontolife.com once via cloudscraper. See header comment in the file — re-running has real security & ethical implications.
- `data/articles.json` — top scraped articles with real titles + bylines
- `data/cf_events.json` — generated by either fetcher; same schema
- `.env` (gitignored) — Cloudflare credentials

## Article metadata

`data/articles.json` was bootstrapped once by visiting each of the top-10 scraped URLs and extracting `<meta og:title>` and JSON-LD `author` fields. This approach worked for the demo but isn't the right production path — it relies on the same Cloudflare Bot Management we're highlighting.

**Production replacement:** query the internal WordPress REST API (or an SJC internal content service) for article metadata by URL/slug. No public scraping, no bot-protection bypass.

## Data shape

`cf_events.json` matches Cloudflare's GraphQL `httpRequestsAdaptiveGroups` response shape — `data.viewer.zones[].httpRequestsAdaptiveGroups[]` with a `count` and `dimensions: {userAgent, clientRequestPath, date, botCategory}` per row. Both the real fetcher and the synthetic generator produce this same shape so `aggregate.py` consumes either source without changes.

## Known limits

- `fetch_cf_data.py` runs one GraphQL query per known AI-bot UA. CF caps each query at 10,000 result rows. For heavy crawlers (Bytespider, GPTBot, ClaudeBot, Amazonbot) the cap is hit — meaning hit counts are **conservative**. Production version would paginate.
- Anonymization is a display-time choice in `aggregate.py`. Real bylines are still present in `data/articles.json`. Treat that file as confidential until the names question is settled.
